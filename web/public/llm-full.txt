# mailrs API - Full Reference

Base URL: https://mail.golia.ai
Auth: All requests require `Authorization: Bearer <API_KEY>` header.

## 1. MCP Integration (recommended for AI agents)

```json
  "mcpServers": {

}
```

### Greylist Local Lists (Phase 2)

Server-wide local white/black overrides on top of the Phase 1 remote sender-domain whitelist. Precedence: any local-black hit beats any local-white hit beats the remote whitelist. A single (kind, value) can only exist on one list at a time (schema-enforced).

| Tool | Permission | Description |
|------|-----------|-------------|
| list_greylist_local | admin.greylist | List local greylist entries; filter by kind (domain/email/cidr) and/or list (white/black) |
| greylist_local_add | admin.greylist | Add a local greylist entry. White = skip greylist, Black = reject 550 |
| greylist_local_remove | admin.greylist | Remove a local greylist entry by id |

REST equivalents:
```
curl -H "Authorization: Bearer <API_KEY>" \
     "https://mail.golia.ai/api/admin/greylist/local-lists?kind=domain&list=white"

curl -X POST -H "Authorization: Bearer <API_KEY>" -H "Content-Type: application/json" \
     -d '{"kind":"domain","list":"white","value":"example.com"}' \
     https://mail.golia.ai/api/admin/greylist/local-lists

curl -X DELETE -H "Authorization: Bearer <API_KEY>" \
     https://mail.golia.ai/api/admin/greylist/local-lists/42
```

Error responses:
- 400 — invalid kind/list/value (e.g. non-canonical CIDR like `10.0.0.1/8` → use `10.0.0.0/8`)
- 409 — `(kind, value)` already exists on the other list; remove first, then re-add
- 403 — missing `admin.greylist` permission

## 2. Permission System

mailrs uses group-based RBAC (role-based access control).

### All Permissions

| Permission | Description |
|-----------|-------------|
| mail.send | Send emails |
| mail.read | Read own mailbox |
| mail.read_domain | Read all mailboxes in a domain |
| admin.domains | Manage domains (add, remove) |
| admin.accounts | Manage accounts (create, remove, password reset) and apps |
| admin.aliases | Manage email aliases and forwarding rules |
| admin.groups | Manage permission groups and memberships |
| admin.queue | Manage outbound delivery queue (view, retry) |
| admin.sieve | Manage sieve filtering scripts |
| admin.impersonate | Send as other users (set arbitrary `from` address) |
| admin.oauth_clients | Manage OAuth client registrations |
| admin.system_config | Edit runtime system config |
| admin.cache | Flush server caches (conversations) |
| admin.greylist | Manage local greylist white/black lists (Phase 2) |
| internal.rpc | Internal RPC calls between services |

### Groups

- **Global groups** (domain = null): permissions apply across all domains
- **Domain groups** (domain = "golia.jp"): permissions scoped to that domain
- **Built-in groups**: `super` (global, all permissions) and `user` (per-domain, mail.send + mail.read)

### Effective Permission Calculation

```
effective = (union of all group permissions) + account grants - account revokes
```

If account is in the `super` group, all permissions on all domains are granted.

### Onboarding Workflow

1. `create_account` -- creates account with password, auto-joins domain's `user` group
2. `add_account_to_group` -- optionally add to admin or custom groups
3. Account can now log in and send/receive email

### Offboarding Workflow

1. `remove_account` -- removes account, group memberships, API keys, and permission overrides (cascade-deleted via FK)

## 3. REST API

### Auth

```
# login (returns session token)
curl -X POST https://mail.golia.ai/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"address": "user@golia.jp", "password": "..."}'

# get current user info
curl https://mail.golia.ai/api/auth/me \
  -H "Authorization: Bearer <token>"

# logout
curl -X POST https://mail.golia.ai/api/auth/logout \
  -H "Authorization: Bearer <token>"
```

### Send Email

```
curl -X POST https://mail.golia.ai/api/mail/send \
  -H "Authorization: Bearer mlrs_..." \
  -H "Content-Type: application/json" \
  -d '{
    "to": ["recipient@example.com"],
    "subject": "Hello",
    "text_body": "Email body text",
    "html_body": "<p>Optional HTML</p>",
    "cc": ["cc@example.com"],
    "reply_to_thread_id": "optional-thread-id"
  }'
```

Response: `{"success": true}`

### Send Email with Attachments (multipart)

```
curl -X POST https://mail.golia.ai/api/mail/send-multipart \
  -H "Authorization: Bearer mlrs_..." \
  -F "to=recipient@example.com" \
  -F "subject=Hello with attachment" \
  -F "text_body=See attached file" \
  -F "attachments=@/path/to/file.png;type=image/png"
```

```
curl "https://mail.golia.ai/api/conversations?limit=20" \
```

## 4. Webhooks

### Create webhook subscription

```
curl -X POST https://mail.golia.ai/api/agent/webhooks \
  -H "Authorization: Bearer mlrs_..." \
  -H "Content-Type: application/json" \
  -d '{
    "callback_url": "https://your-server.com/webhook",
    "event_type": "new_message",
    "filter_contact": "important@example.com",
    "filter_thread_id": "optional-thread-id"
  }'
```

Response:
```json
{
  "id": 1,
  "signing_secret": "your-hmac-secret"
}
```

### Webhook payload (POST to your callback_url)

Headers:
- `X-Mailrs-Signature`: HMAC-SHA256 hex digest
- `X-Mailrs-Event`: event type
- `Content-Type`: application/json

Body:
```json
{
  "event": "new_message",
  "message_id": "abc@mail.golia.ai",
  "uid": 12345,
  "from": "sender@example.com",
  "subject": "New email subject",
  "timestamp": 1710000000
}
```

### Verify signature

```
HMAC-SHA256(signing_secret, request_body) == X-Mailrs-Signature
```

Payload contains metadata only. Call `GET /api/mail/messages/{uid}` for full content.
Failed deliveries retry with exponential backoff (60s -> 6h, max 8 attempts).

### List webhooks

```
curl https://mail.golia.ai/api/agent/webhooks \
  -H "Authorization: Bearer mlrs_..."
```

### Delete webhook

```
curl -X DELETE https://mail.golia.ai/api/agent/webhooks/1 \
  -H "Authorization: Bearer mlrs_..."
```

## 5. API Keys & Apps

### User API Keys

User API keys are bound to a specific account and inherit that account's permissions. They must be created via session auth (web login).

```
# create key (requires session auth)
curl -X POST https://mail.golia.ai/api/agent/keys \
  -H "Authorization: Bearer <session_token>" \
  -H "Content-Type: application/json" \
  -d '{"name": "my-agent", "expires_at": "2027-01-01T00:00:00Z"}'
```

Response:
```json
{
  "id": 1,
  "key": "mlrs_625735a4_631d4f9a29061c279419fb17b3939f9d7eebfba8",
  "prefix": "625735a4",
  "name": "my-agent",
  "expires_at": "2027-01-01T00:00:00Z"
}
```

```
# list keys
curl https://mail.golia.ai/api/agent/keys \
  -H "Authorization: Bearer <session_token_or_api_key>"

# revoke key
curl -X DELETE https://mail.golia.ai/api/agent/keys/1 \
  -H "Authorization: Bearer <session_token>"
```

Revocation is immediate. Cached credentials are evicted from Valkey.

### App API Keys

App keys are created when registering an app via `POST /api/admin/apps` or the MCP `create_app` tool. They have explicit scopes (comma-separated permission names) rather than inheriting from a user account.

Scopes use the same permission names as the RBAC system (e.g., `mail.send,mail.read,admin.aliases`).

### internal.rpc

The `internal.rpc` permission is reserved for service-to-service communication. It is not grantable to regular users. App keys with this scope can make internal RPC calls between mailrs components.

## 6. Auth Notes

- API key format: `mlrs_{8-char prefix}_{40-char secret}` (54 chars total)
- Keys are SHA-256 hashed in DB, cached in Valkey for fast lookup (5min TTL)
- Admin keys (with admin.accounts/admin.groups permissions) can manage accounts and groups
- Super users can set `from` to any address across all managed domains
- Regular keys can only send as the associated account
- Session tokens are obtained via `POST /api/auth/login` with `{"address":"...","password":"..."}`
- Both session tokens and API keys are accepted in the `Authorization: Bearer` header

## 7. Error Responses

All errors return JSON:
```json
{"error": "description of the error"}
```

HTTP status codes:
- 400 -- bad request (invalid input, missing required fields)
- 401 -- authentication required or invalid credentials
- 403 -- forbidden (insufficient permissions for the requested operation)
- 404 -- not found (message, account, domain, alias, or app does not exist)
- 429 -- rate limited (back off and retry)
- 500 -- server error (internal failure)
