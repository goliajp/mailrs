   - Find the block where `from != &user` returns error

   - If superadmin check passes, allow the send
   - If not superadmin or domain not in super_domains, return existing error
   - The AuthUser destructuring already extracts `super_domains` (established in Phase 1)

3. **Apply same from-validation to `send_message_multipart`:**
   - Find the equivalent from-validation block in the multipart handler
   - Apply identical superadmin logic (extract to a helper function if both handlers share the pattern, or inline if the code is short)

4. **Implement reply_to_thread_id resolution in `send_message`:**
   - If `reply_to_thread_id` is Some and `in_reply_to` is None:
     - Query the thread's messages to find the last message's Message-ID (use existing mailbox store / conversations query)
     - Set `in_reply_to` to that Message-ID
     - Build References header from the thread (use existing `get_thread_references` if available, or query messages in thread ordered by date)
   - If both `reply_to_thread_id` and `in_reply_to` are provided, `in_reply_to` takes precedence (explicit wins)

5. **Apply reply_to_thread_id to multipart handler:** Check if the multipart handler's form fields include in_reply_to. If yes, add reply_to_thread_id as an optional form field with the same resolution logic. If multipart handler does not support in_reply_to, add both fields. Multipart replies must work the same as JSON replies.

IMPORTANT: Do NOT create new route paths. Modify existing handlers only. Use immutable patterns — create new structs/values rather than mutating existing ones.
