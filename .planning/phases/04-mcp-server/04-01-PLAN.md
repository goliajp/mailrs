   - Define `MailMcpService` struct with `web_state: Arc<WebState>` and `auth_user: AuthUser` and `tool_router: ToolRouter<Self>`
   - Implement `#[tool_router] impl MailMcpService` with a `new(web_state, auth_user)` constructor that calls `Self::tool_router()`
   - Implement `#[tool_handler] impl ServerHandler for MailMcpService` with `get_info()` returning:
     - `protocol_version: ProtocolVersion::V_2025_03_26`
     - `capabilities: ServerCapabilities::builder().enable_tools().build()`
     - `server_info: Implementation { name: "mailrs", version: env!("CARGO_PKG_VERSION") }`
     - `instructions: Some("mailrs MCP server — tools for sending, reading, searching, and replying to emails.")`
   - Create `pub fn setup_mcp(state: Arc<WebState>) -> axum::Router` that:
     - Creates `StreamableHttpService::new()` with factory closure
     - The factory closure extracts nothing yet (auth handled by middleware before)
     - Uses `LocalSessionManager::default().into()`
     - Returns `axum::Router::new().nest_service("/mcp", service)`
   - NOTE on auth integration: The factory closure receives no per-request auth info. Instead, `mcp_auth_middleware` in auth.rs will validate the Bearer token and inject `AuthUser` into request extensions BEFORE the request reaches the MCP service. Inside each tool method, access `self.auth_user` which was set during construction. Since the factory creates one service instance per session, we need to pass auth info through. Two approaches:
     a) Use rmcp's `on_request_fn` to extract auth from request headers on each request
     b) Create a wrapper that validates auth in the middleware and stores it in request extensions, then have the MCP service read from extensions on `initialize`
     Choose approach (a) if rmcp supports `on_request_fn` for streamable HTTP, otherwise use approach (b) — the middleware sets a header/extension, and on first MCP initialize, the service reads auth. If neither is clean, fall back to: each tool method manually calls `api_key_store::verify_api_key_raw()` using the token stored in the service instance (set via factory closure capturing the token from request headers). Document which approach was chosen.
   - Add `mod mcp;` to `crates/server/src/main.rs` (or wherever modules are declared)

4. Create `crates/server/src/mcp/auth.rs`:
   - Implement `pub(crate) async fn mcp_auth_middleware(...)` as axum middleware
   - Extract `Authorization: Bearer <token>` header
   - If token starts with `mlrs_`, call the same API key verification logic used in `web/auth.rs` (reuse `api_key_store::verify_by_raw_key` or similar)
   - On success, insert `AuthUser` into request extensions
   - On failure, return 401 Unauthorized
   - This middleware will be applied as a layer on the MCP router in Plan 02's wiring

Create `crates/server/src/mcp/tools.rs` with 5 tool parameter structs and implementations.

**Parameter structs** (all derive `Debug, Deserialize, schemars::JsonSchema`):

1. `SendEmailParams`:
   - `from: Option<String>` — `#[schemars(description = "Sender email address. Omit to use authenticated account.")]`
   - `to: Vec<String>` — `#[schemars(description = "Recipient email addresses")]`
   - `cc: Option<Vec<String>>` — `#[schemars(description = "CC recipients")]`
   - `subject: String`
   - `body: String` — `#[schemars(description = "Plain text email body")]`
   - `html_body: Option<String>`

2. `ReadEmailParams`:
   - `uid: u32` — `#[schemars(description = "Message UID from list_conversations or search_emails results")]`

3. `SearchEmailsParams`:
   - `query: String` — `#[schemars(description = "Search query string")]`
   - `limit: Option<u32>` — `#[schemars(description = "Max results (default 20, max 20)")]`

4. `ReplyEmailParams`:
   - `thread_id: String` — `#[schemars(description = "Thread ID to reply to (from list_conversations)")]`
   - `body: String` — `#[schemars(description = "Reply text body")]`
   - `from: Option<String>`

5. `ListConversationsParams`:
   - `limit: Option<u32>` — `#[schemars(description = "Max results (default 20, max 20)")]`
   - `category: Option<String>` — `#[schemars(description = "Filter by category: personal, notification, promotion, general")]`

**Tool implementations** (add as `#[tool]` methods on `MailMcpService` in mod.rs's `#[tool_router] impl` block):

All tools access `self.web_state` for `mailbox_store`, `outbound_queue`, etc. and `self.auth_user` for address/permissions.

1. `#[tool(description = "Send an email. Returns message ID on success.")]`
   `async fn send_email(&self, #[tool(aggr)] params: SendEmailParams) -> Result<CallToolResult, McpError>`
   - Validate: `to` not empty, total recipients <= 50
   - Resolve `from`: use `params.from` or fall back to `self.auth_user.address`
   - Call `verify_sender(from, &self.auth_user.address, &self.auth_user.super_domains)` — on error return `McpError::invalid_params`
   - Build RFC5322 message using same pattern as `web::mail::send_message` (call `build_rfc5322_message` or `build_rfc5322_with_attachments` with empty attachments)
   - Queue via `deliver_message` pattern (enqueue to outbound_queue, save to maildir)
   - Return `CallToolResult::success` with JSON `{ "message_id": "...", "status": "queued" }`
   - NOTE: The send logic in web/mail.rs is tightly coupled to axum handlers. Rather than extracting it, replicate the core flow: verify_sender → generate message_id → resolve_thread_reply (if needed) → build_rfc5322_with_attachments → deliver_message. The `deliver_message` and `build_rfc5322_with_attachments` functions are `pub(super)` — you'll need to change them to `pub(crate)` so MCP can call them. Same for `resolve_thread_reply` if used.

2. `#[tool(description = "Read an email by UID. Returns sender, subject, text body, and metadata.")]`
   `async fn read_email(&self, #[tool(aggr)] params: ReadEmailParams) -> Result<CallToolResult, McpError>`
   - Get mailbox_store, iterate mailboxes, find message by uid (same pattern as `web::mail::get_message`)
   - Return JSON with: sender, subject, text_body (NOT html_body — AI doesn't need HTML), internal_date, message_id, category
   - Return `McpError::invalid_params("message not found")` if uid doesn't exist

3. `#[tool(description = "Search emails by keyword. Returns conversation summaries (thread_id, subject, snippet, participants).")]`
   `async fn search_emails(&self, #[tool(aggr)] params: SearchEmailsParams) -> Result<CallToolResult, McpError>`
   - Limit hardcoded to min(params.limit.unwrap_or(20), 20) — prevent token overflow
   - Call `mailbox_store.search_conversations(user, query, limit, None, None)`
   - Map results to simplified JSON: `[{ thread_id, subject, snippet, participants, last_date, message_count }]`
   - Do NOT include semantic search supplement (keep it simple for MCP)

4. `#[tool(description = "Reply to an email thread. Automatically sets In-Reply-To headers. Returns message ID.")]`
   `async fn reply_email(&self, #[tool(aggr)] params: ReplyEmailParams) -> Result<CallToolResult, McpError>`
   - Resolve from (same as send_email)
   - Call resolve_thread_reply or replicate: lookup last message_id in thread via `mailbox_store.get_last_message_id_in_thread()`
   - If thread not found, return McpError
   - Build reply with in_reply_to + references
   - Auto-populate subject from thread: look up thread, prepend "Re: " if not already
   - Determine recipients: reply to original sender (the sender of the last message in thread)
   - Queue and return message_id

5. `#[tool(description = "List recent email conversations. Returns thread summaries (thread_id, subject, snippet, participants, unread count).")]`
   `async fn list_conversations(&self, #[tool(aggr)] params: ListConversationsParams) -> Result<CallToolResult, McpError>`
   - Limit hardcoded to min(params.limit.unwrap_or(20), 20)
   - Call `mailbox_store.list_conversations(user, limit, None, category, None, false, None)`
   - Map to simplified JSON: `[{ thread_id, subject, snippet, participants, message_count, unread_count, last_date, category }]`
