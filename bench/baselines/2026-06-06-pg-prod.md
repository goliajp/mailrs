# API e2e baseline — PG-side prod, 2026-06-06

Pre-SPG-cutover baseline. After cutover to `goliakk/spg:X.Y.Z` we re-run
`bench/api-e2e.ts` against the same `https://mail.golia.ai` endpoint and
diff against the numbers below.

## Environment

- **base**: `https://mail.golia.ai`
- **mailrs version**: `v1.7.102` (commit `551b423`)
- **DB**: PostgreSQL 18 + pgvector 0.8.2 (`pgvector/pgvector:0.8.2-pg18`)
- **kevy**: in-process embed (v9 Phase C, kevy-embedded 1.1.2)
- **client location**: macOS workstation (doracawl) — public internet
- **network baseline (curl, 5 cold + warm GET to `/api/health`)**:
  - DNS: ~2 ms (cached) / 19 ms (cold)
  - TCP: 12-13 ms
  - TLS handshake: 13 ms incremental
  - TTFB: 36 ms
  - Total: 36 ms
- **samples per endpoint**: 5
- **auth**: bearer JWT for `lihao@golia.jp`

User-reported pain: "mark as read 非常非常慢，点快了还会失败". Server-side

HtmlFrame sanitize chain — fixed in `d7f5670`.

## Conclusion (pre-cutover state)

- Server-side latency is **not the bottleneck for any user-visible path**
  at the current PG + kevy stack.
- The user-reported 2-3 s thread-open lag was a frontend mutation /
  sanitize / re-render cascade (fixed in commit `d7f5670`).
- Cold list fetch 325 ms is dominated by TLS handshake (24 ms) + cache
  miss; warm 50 ms. Acceptable.
- All thread fetches P95 ≤ 160 ms across 50-thread sample; biggest 73 KB
  body steady at 60 ms.
- Mark-read 40 ms median, no failures observed in 30-call test.
